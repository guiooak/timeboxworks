Choosing a datetime library

A hands-on benchmark of Luxon, date-fns, Day.js — plus native Date and Moment for reference — across bundle size, raw speed, interval logic, and DST correctness.

📦 Bundle size

Tree-shaken & minified, bundling a realistic time-tracker entrypoint (parse + diff + duration format + timestamp format). Lower is better.

⚠️ Moment shown bare-core; with locales (default) it commonly balloons to 60–70 KB gzip. Day.js / date-fns / Luxon all use the browser's built-in Intl for timezones, shipping no extra TZ data.

Operations per second on a warm JIT (tinybench). Higher is better. Native wins everywhere — but note the absolute numbers: even the slowest library is sub-millisecond.

🔗 Interval stress test

The operations a time tracker actually leans on: detecting overlapping sessions, splitting a session at calendar midnight for per-day totals, and summing many sessions. This is where the rankings reshuffle.

🕑 DST correctness

Physical elapsed time when a session straddles a daylight-saving transition, given wall-clock local times in America/New_York. Correctness, not speed — and the one place "no library" genuinely breaks.

Every real library is correct (all delegate to Intl/IANA). Native naive returns the wall-clock delta — the classic billing bug. Even a careful ~18-line hand-rolled Intl version is still wrong at the boundary (single-pass offset resolution), which is the strongest argument against rolling your own.

🏆 Verdict

Your app's core entity is the interval / duration — and that's where the headline-speed story flips.

Concrete suggestion

Store timestamps as epoch millis (or UTC ISO), use date-fns for parsing / formatting / comparison, and pull in date-fns-tz only where you cross timezones. Near-native size and speed, correct and readable interval logic.

Reproduce it yourself: cd scripts && npm i luxon dayjs date-fns date-fns-tz moment moment-timezone tinybench esbuild && node bench.mjs
